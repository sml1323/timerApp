# Product Requirements Document - bmad_test

**Author:** Imseungmin
**Date:** 2026-03-15 23:38:43

## Executive Summary

bmad_test는 자기주도 학습자를 위한 macOS 기반 학습 운영 앱이다. 이 제품의 목적은 단순한 포모도로 타이머 제공이 아니라, 사용자가 주제별 학습 시간을 구조적으로 기록하고 주간 목표 대비 실제 진척을 명확하게 확인하도록 만들어 학습의 꾸준함을 유지하게 하는 것이다.

핵심 문제는 사용자가 공부를 하더라도 어떤 주제에 얼마나 시간을 썼는지, 이번 주 목표 대비 얼마나 진행했는지를 일관되게 파악하기 어렵다는 점이다. 일반 타이머 앱은 시간 측정에는 도움이 되지만 목표 추적과 주제별 기록이 약하고, 수기 기록 방식은 유연하지만 번거롭고 즉시성이 떨어진다. 이 때문에 사용자는 하루 이틀 계획이 어긋나면 쉽게 흐름을 잃고, 실제로는 학습을 이어가고 있어도 성취감을 체계적으로 느끼지 못한다.

이 제품은 기본 포모도로 사이클, 주제 생성 및 선택, 세션 기록, 주제별 주간 목표 설정, 진행 통계를 하나의 흐름으로 묶어 이 문제를 해결한다. 사용자는 공부 시작 전에 주제를 선택하고 세션을 시작하며, 종료 후 자동으로 누적 시간과 목표 대비 남은 시간을 확인할 수 있다. 이를 통해 사용자는 단순히 시간을 재는 것이 아니라, 자신의 학습 루틴을 운영하고 있다는 감각을 얻게 된다.

### What Makes This Special

이 제품의 차별점은 하루 단위의 실패 여부보다 주간 누적 진척을 중심으로 학습 경험을 설계한다는 점이다. 사용자는 하루를 완벽히 채우지 못했더라도, 주간 목표 관점에서 다시 이어갈 수 있다. 이 구조는 죄책감보다 지속성을 강화하고, 사용자가 장기적인 학습 루틴을 유지하도록 돕는다.

또한 bmad_test는 타이머, 기록, 목표, 통계를 분리된 기능으로 제공하지 않고 하나의 학습 운영 흐름으로 통합한다. 사용자는 어떤 주제를 얼마나 했는지, 목표까지 얼마나 남았는지, 이번 주에 실제로 잘하고 있는지를 즉시 확인할 수 있다. 이는 일반 포모도로 앱이나 수기 기록 도구 대비 더 강한 성취감과 동기부여를 제공한다.

핵심 인사이트는 사람들에게 더 많은 학습 의지가 없는 것이 아니라, 자신의 진척이 구조적으로 보이지 않기 때문에 꾸준함을 유지하기 어렵다는 점이다. 이 제품은 그 공백을 메우는 데 집중한다.

## Success Criteria

### User Success

사용자의 가장 중요한 성공 순간은 이번 주 목표 달성률과 누적 학습 진척이 명확하게 보일 때다. 이 앱은 사용자가 단순히 타이머를 돌렸는지가 아니라, 이번 주에 실제로 얼마나 해냈는지를 이해하게 만들어야 한다.

사용자는 다음과 같은 상태에 도달했을 때 제품 가치를 체감한다.

- 이번 주 목표 대비 현재 진행률을 즉시 확인할 수 있다.
- 어떤 주제에 시간을 얼마나 썼는지 구조적으로 이해할 수 있다.
- 하루 계획이 어긋나더라도 주간 기준으로 다시 이어갈 수 있다.
- 자신의 학습 루틴이 유지되고 있다는 성취감과 동기부여를 얻는다.

### Business Success

이 제품은 초기에는 개인용 앱이므로 매출, 팀 협업, 조직 도입보다 개인 사용자 가치 검증이 우선이다. 사업적 성공은 시장 확장보다 개인 사용자 1명에게도 반복 사용 가치가 명확한지로 판단한다.

초기 성공 기준은 다음과 같다.

- 사용자가 최소 1주 동안 실제 학습 루틴에 앱을 사용한다.
- 사용자가 주간 목표 추적 기능을 반복적으로 활용한다.
- 사용자가 통계 화면을 통해 자신의 진척을 점검하는 습관을 만든다.
- 사용자가 이 앱을 단순 타이머보다 더 유용한 학습 운영 도구로 인식한다.

### Technical Success

기술적으로 이 제품은 신뢰 가능한 개인 학습 기록 도구여야 한다. 따라서 다음 조건이 반드시 충족되어야 한다.

- 앱 재실행 후에도 세션 기록, 주제 정보, 주간 목표 데이터가 보존되어야 한다.
- 타이머는 백그라운드 상태에서도 정확하게 유지되어야 한다.
- 세션 완료 후 통계와 목표 진행률이 즉시 반영되어야 한다.
- 기본 사용 흐름에서 데이터 손실이나 기록 불일치가 없어야 한다.

### Measurable Outcomes

초기 단계에서 추적해야 할 측정 가능한 결과는 다음과 같다.

- 주간 목표 달성률
- 주간 총 학습 시간
- 주제별 누적 학습 시간
- 통계 화면 조회 빈도
- 주간 세션 실행 횟수
- 최소 1주 연속 실제 사용 여부

- JSON export
- 월간/연간 통계
- 태그별 상세 분석

### Vision (Future)

장기적으로는 macOS 개인 학습 운영 앱으로 완성도를 높이는 것이 우선이다. 이후 필요 시 더 깊은 통계, 확장된 분류 체계, 추가 플랫폼 연동 같은 기능을 검토할 수 있지만, 현재 단계에서는 개인 사용자의 꾸준함 유지라는 핵심 가치에 집중한다.

## User Journeys

### Journey 1: Primary User - Core Success Path

민준은 저녁 공부를 시작하기 전에 bmad_test를 연다. 앱을 실행하면 가장 먼저 오늘 진행 상황과 이번 주 목표 대비 현재 진척을 확인할 수 있다. 그는 오늘 얼마나 했는지와 이번 주 목표까지 얼마나 남았는지를 빠르게 파악한 뒤, 지금 진행할 학습 주제를 선택한다.

주제를 고른 뒤 민준은 포모도로 타이머를 시작한다. 학습 세션 동안 앱은 불필요한 복잡함 없이 집중 상태를 유지하게 하고, 세션이 끝나면 해당 시간이 자동으로 기록된다. 세션 종료 후 그는 오늘 누적 시간과 주제별 누적 시간이 즉시 갱신된 것을 본다.

이후 민준은 통계 탭으로 이동해 이번 주 전체 진척과 주제별 학습 분포를 확인한다. 이 순간 제품의 핵심 가치가 전달된다. 그는 단순히 시간을 보낸 것이 아니라, 이번 주 목표를 향해 실제로 전진하고 있다는 감각을 얻는다.

이 여정은 다음 요구사항을 드러낸다:
- 앱 첫 화면에서 오늘 진행 상황과 주간 진척을 함께 보여줘야 함
- 사용자는 빠르게 주제를 선택하고 타이머를 시작할 수 있어야 함
- 세션 완료 후 기록과 통계가 즉시 갱신되어야 함
- 주제 관리와 통계 화면으로 이동하는 명확한 탭 구조가 필요함

### Journey 2: Primary User - Interrupted or Incomplete Day

민준은 오늘도 공부를 시작하려고 앱을 열지만, 예상보다 시간이 부족하다. 그는 앱에서 오늘 진행 상황을 먼저 확인하고, 짧게라도 한 세션을 진행할지 판단한다. 주제를 선택해 타이머를 시작하지만, 중간에 급한 일이 생겨 세션을 끝까지 완료하지 못할 수도 있다.

이 상황에서 중요한 것은 앱이 사용자를 실패한 사람처럼 느끼게 하지 않는 것이다. 민준은 오늘 목표를 다 채우지 못했더라도, 주간 목표 기준으로 아직 얼마나 남았는지 확인할 수 있어야 한다. 하루가 어긋나도 이번 주 전체 흐름 안에서 다시 이어갈 수 있다는 점이 중요하다.

세션 중단이나 목표 미달 상황에서도 앱은 현재 기록 상태를 일관되게 보여주고, 다음에 다시 시작할 수 있는 기준점을 제공해야 한다. 사용자는 "오늘 망했다"가 아니라 "이번 주 기준으로 다시 이어가면 된다"는 해석을 받아야 한다.

이 여정은 다음 요구사항을 드러낸다:
- 오늘 목표 미달 상황에서도 주간 진척을 계속 보여줘야 함
- 세션 중단 또는 미완료 상태를 사용자에게 혼란 없이 표현해야 함
- 실패감이 아니라 재시작 가능성을 강화하는 피드백 문구와 정보 구조가 필요함

### Journey 3: Primary User - Wrong Topic or Course Correction

민준은 공부를 시작하며 주제를 선택했지만, 세션을 진행하다 보니 실제로는 다른 주제로 공부하고 있었다는 것을 깨닫는다. 또는 오늘 계획했던 과목보다 더 급한 주제를 먼저 해야 한다고 판단한다.

이 경우 사용자는 현재 세션이나 기록을 어떻게 다뤄야 할지 명확히 이해할 수 있어야 한다. 잘못 선택한 주제를 유지한 채 기록이 쌓이면 통계 신뢰도가 떨어지고, 결국 앱 전체 신뢰도도 낮아진다. 반대로 수정이 너무 어렵거나 번거로우면 사용자는 다시 수기 기록으로 돌아갈 가능성이 높다.

따라서 사용자는 주제를 바꾸거나, 필요한 경우 기록을 정정할 수 있어야 하며, 앱은 통계를 신뢰 가능한 상태로 유지해야 한다. 개인 학습 앱에서 기록의 정확성은 동기부여의 전제 조건이다.

이 여정은 다음 요구사항을 드러낸다:
- 잘못 선택한 주제를 정정하거나 변경할 수 있는 흐름이 필요함
- 세션 기록 수정 시 통계가 일관되게 재계산되어야 함
- 기록 정정은 빠르고 단순해야 하며 사용자 부담이 낮아야 함

### Journey 4: Navigation and Review Loop

민준은 어떤 날은 타이머를 바로 시작하지 않고, 먼저 앱의 다른 영역을 확인한다. 사이드 탭 또는 명확한 내비게이션을 통해 주제 생성 화면으로 이동해 새 주제를 만들거나 기존 주제를 정리할 수 있어야 한다. 또 다른 날에는 통계 화면으로 바로 이동해 오늘, 이번 주, 주제별 진행 상황을 먼저 점검하고 공부 계획을 세운다.

이 여정에서 중요한 것은 앱이 타이머 화면 하나만 있는 도구가 아니라는 점이다. 사용자는 실행 전 계획, 실행 중 집중, 실행 후 검토를 모두 지원받아야 한다. 따라서 주제 관리, 오늘 진행 상황, 기본 통계는 하나의 연결된 운영 흐름 안에 있어야 한다.

이 여정은 다음 요구사항을 드러낸다:
- 오늘 진행 상황을 중심으로 한 홈 또는 메인 화면이 필요함
- 주제 생성/관리 화면이 독립된 내비게이션 항목으로 필요함
- 통계 화면이 독립된 내비게이션 항목으로 필요함
- 사용자는 실행 전과 실행 후 모두 앱에서 의미 있는 결정을 할 수 있어야 함

### Journey Requirements Summary

위 사용자 여정은 다음 핵심 기능군을 요구한다.

- 메인 화면에서 오늘 진행 상황과 주간 진척을 함께 보여주는 대시보드
- 주제 선택 후 즉시 타이머를 시작할 수 있는 핵심 학습 흐름
- 세션 완료 후 자동 기록 저장 및 즉시 갱신되는 통계
- 주제 생성 및 관리 기능
- 통계 화면과 메인 화면 사이를 오갈 수 있는 단순한 탭 또는 사이드 내비게이션
- 목표 미달, 세션 중단, 기록 정정 같은 예외 상황을 다루는 회복 가능한 UX
- MVP 범위에서는 태그 기능을 제외하고, 이후 확장 가능 항목으로 유지

## Domain-Specific Requirements

### Compliance & Regulatory

이 제품은 edtech 도메인에 속하지만, 초기 MVP는 개인용 macOS 앱으로 설계되며 학교 시스템 연동, 아동 계정, 다중 사용자 관리, 교육기관 배포를 포함하지 않는다. 따라서 COPPA, FERPA 같은 교육 규제의 직접 적용 범위는 MVP 기준에서 핵심 제약으로 보지 않는다.

다만 학습 기록과 주제 정보는 개인 사용자 활동 데이터이므로, 데이터 처리 방식은 사용자 신뢰를 해치지 않도록 단순하고 명확해야 한다. 제품은 불필요한 개인정보 수집을 피하고, 계정 기반 식별 없이 개인 기기 내 데이터 관리에 집중해야 한다.

### Technical Constraints

제품은 로컬 우선 저장 방식을 따른다. 세션 기록, 주제 정보, 주간 목표, 통계 계산에 필요한 데이터는 기본적으로 사용자의 macOS 기기 내부에 저장되어야 하며, 서버 기반 계정 시스템이나 클라우드 동기화를 MVP 전제 조건으로 두지 않는다.

기술적으로 다음 제약이 중요하다.

- 로그인 및 계정 시스템은 MVP에 포함하지 않는다.
- 앱 재실행 이후에도 모든 핵심 데이터가 일관되게 유지되어야 한다.
- 통계 계산은 로컬 데이터 기준으로 즉시 반영되어야 한다.
- 타이머 상태와 세션 기록은 사용자 신뢰를 해치지 않을 정도로 안정적으로 유지되어야 한다.
- 기본적인 macOS 접근성 기준을 충족해야 한다.

### Integration Requirements

MVP 범위에서는 외부 교육 플랫폼, LMS, 캘린더, 클라우드 저장소, 학교 시스템과의 연동을 요구하지 않는다. 제품은 독립 실행형 개인 학습 앱으로 동작해야 하며, 핵심 가치는 외부 시스템 의존 없이 완결되어야 한다.

향후 확장 시 고려할 수 있는 통합은 존재하지만, 현재 단계에서는 내부 데이터 모델과 로컬 사용 흐름만으로도 제품 가치가 성립해야 한다.

### Risk Mitigations

이 도메인에서 가장 큰 위험은 규제가 아니라 사용자 신뢰 상실이다. 특히 학습 기록, 누적 시간, 목표 달성률이 부정확하게 표시되면 제품의 핵심 가치가 무너진다. 이를 줄이기 위해 다음 원칙이 필요하다.

- 세션 기록과 통계 계산 결과가 항상 일관되게 유지되어야 한다.
- 주제 변경, 기록 정정, 세션 중단 같은 예외 상황에서도 데이터가 왜곡되지 않아야 한다.
- 사용자는 자신의 데이터가 계정이나 외부 서버 없이 로컬에 저장된다는 점을 이해할 수 있어야 한다.
- 접근성은 최소 기준을 만족해 기본적인 사용 장벽이 없도록 해야 한다.

## Desktop App Specific Requirements

### Project-Type Overview

bmad_test는 macOS 전용 개인 학습 앱으로 설계된다. 제품의 핵심 가치는 빠른 실행, 로컬 중심 데이터 보존, 방해 없는 학습 흐름 제공에 있다. 따라서 브라우저 기반 경험이나 다중 플랫폼 추상화보다, macOS 환경에서 자연스럽고 안정적으로 동작하는 데 우선순위를 둔다.

### Technical Architecture Considerations

제품은 macOS 단일 플랫폼을 대상으로 하며, 첫 버전에서는 Windows나 웹 버전을 고려하지 않는다. 기능 설계와 UI 구조는 macOS 데스크톱 사용 패턴에 맞춰 최적화되어야 한다.

앱은 완전 오프라인 상태에서도 핵심 기능이 동작해야 한다. 타이머 실행, 세션 기록, 주제 관리, 주간 목표 추적, 기본 통계 확인은 네트워크 연결 없이도 모두 가능해야 한다. 이는 로컬 우선 저장 전략과 일치해야 한다.

업데이트 시스템은 MVP 범위에서 필수 요구사항으로 두지 않는다. 첫 버전에서는 자동 업데이트 메커니즘 없이도 배포 가능해야 하며, 이후 제품 안정화 단계에서 도입 여부를 검토한다.

### Platform Support

- 지원 플랫폼은 macOS 단일 플랫폼으로 제한한다.
- 첫 버전에서는 웹, 모바일, Windows 지원을 요구하지 않는다.
- 기능 우선순위와 UX 의사결정은 macOS 사용 흐름에 맞춰야 한다.

### System Integration

MVP에서 필요한 시스템 연동은 최소 수준으로 유지한다. 다만 학습 세션의 시작과 종료, 휴식 전환, 진행 상기 등을 위해 macOS 알림(Notification)은 유용한 보조 기능으로 고려한다.

현재 단계에서 필수 연동으로 보지 않는 항목은 다음과 같다.

- 메뉴바 상주 앱 구조
- 글로벌 단축키
- 로그인 시 자동 실행
- 복잡한 Dock 상호작용
- 시스템 캘린더 또는 외부 앱 연동

알림은 있으면 좋은 기능으로 보되, 앱 핵심 가치가 알림 의존적으로 설계되어서는 안 된다.

### Offline Capabilities

- 인터넷 연결 없이 모든 핵심 기능이 동작해야 한다.
- 세션 기록과 통계 계산은 로컬 데이터만으로 처리되어야 한다.
- 네트워크 연결 상태는 타이머 정확성이나 기록 저장에 영향을 주지 않아야 한다.

### Implementation Considerations

이 제품은 가벼운 개인용 데스크톱 앱으로 느껴져야 한다. 앱 실행 후 빠르게 오늘 진행 상황을 확인하고, 주제를 선택하고, 타이머를 시작할 수 있어야 한다. 데스크톱 앱 특성상 사용자 기대치는 "언제 열어도 즉시 쓸 수 있는 도구"에 가깝기 때문에, 초기 로딩 지연, 복잡한 설정 플로우, 네트워크 의존 초기화는 피해야 한다.

또한 알림 기능을 도입하더라도 사용자의 집중을 돕는 수준으로 제한해야 한다. 제품 정체성은 생산성 보조가 아니라 학습 흐름 운영에 있으므로, 시스템 연동은 핵심 흐름을 강화하는 선에서만 사용해야 한다.

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** problem-solving MVP  
이 제품의 MVP는 학습자의 핵심 문제인 "주간 목표 기준으로 꾸준함을 유지하고 실제 진척을 명확히 확인하는 것"을 가장 작은 범위로 해결하는 데 초점을 둔다. 제품의 성공은 기능 수가 아니라, 사용자가 실제로 한 주 동안 이 앱을 써 보며 목표 추적과 학습 지속성에 도움이 된다고 느끼는지로 판단한다.

**Resource Requirements:** 1인 개발  
초기 개발은 1인 개발 전제를 기준으로 계획한다. 따라서 범위는 작고 명확해야 하며, 기능 간 의존성이 낮고 구현 복잡도가 통제 가능해야 한다. 설계 판단은 장기 확장성보다 빠른 검증과 안정적인 핵심 루프 완성에 우선순위를 둔다.

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- 오늘 진행 상황과 주간 진척 확인
- 주제 선택 후 포모도로 세션 시작
- 세션 완료 후 자동 기록 반영
- 기본 통계 화면에서 진행 결과 확인
- 목표 미달 상태에서도 주간 기준으로 다시 이어가기

**Must-Have Capabilities:**
- 메인 화면에서 오늘 진행 상황과 주간 목표 대비 진척 표시
- 주제 생성 및 관리
- 주제 선택 기반 포모도로 타이머 실행
- 세션 완료 기록 저장
- 주제별 주간 목표 설정
- 기본 통계 화면
- 앱 재실행 후 데이터 보존
- 기록 수정 또는 정정 시 통계 재계산의 일관성 보장

### Post-MVP Features

**Phase 2 (Post-MVP):**
- JSON export
- 월간/연간 통계
- 태그 기반 분류 복귀
- 태그별 상세 분석
- 알림 경험 고도화

**Phase 3 (Expansion):**
- 플랫폼 확장 검토
- 클라우드 동기화 또는 백업
- 더 정교한 학습 분석
- 외부 도구 연동
- 고급 개인화 기능

### Risk Mitigation Strategy

**Technical Risks:**  
가장 큰 기술 리스크는 기록과 통계의 신뢰성이 깨지는 것이다. 세션 기록, 목표 진행률, 누적 시간 계산이 조금이라도 어긋나면 제품의 핵심 가치가 무너진다. 따라서 데이터 저장 모델, 통계 계산 규칙, 기록 수정 흐름은 초기부터 단순하고 검증 가능하게 설계해야 한다.

**Market Risks:**  
사용자가 이 앱을 기존 타이머 앱보다 더 가치 있게 느끼지 못하면 유지되지 않는다. 이를 줄이기 위해 MVP는 통계 깊이보다 "오늘 얼마나 했고 이번 주 얼마나 남았는지"를 즉시 이해할 수 있는 경험에 집중해야 한다.

**Resource Risks:**  
1인 개발 프로젝트에서 가장 큰 운영 리스크는 범위 확장이다. 따라서 MVP는 핵심 루프 외의 기능을 의도적으로 배제하고, Post-MVP 후보를 엄격히 분리해야 한다. JSON export, 월간/연간 통계, 태그 상세 분석은 출시 후 단계로 미룬다.

- FR6: 시스템은 완료된 학습 세션을 자동으로 기록할 수 있다.

- FR10: 사용자는 기존 학습 주제 목록을 확인할 수 있다.

- FR18: 사용자는 메인 화면에서 오늘 진행 상황을 확인할 수 있다.
- FR19: 사용자는 메인 화면에서 이번 주 목표 대비 전체 진척을 확인할 수 있다.
- FR20: 사용자는 주제별 누적 학습 시간을 확인할 수 있다.
- FR21: 사용자는 기본 통계 화면에서 전체 학습 진행 결과를 확인할 수 있다.
- FR22: 사용자는 오늘 기준과 주간 기준의 학습 상태를 구분해 확인할 수 있다.

### 기록 정정 및 데이터 신뢰성

- FR26: 시스템은 기록의 주제가 변경되면 관련 통계를 일관되게 재계산할 수 있다.
- FR27: 시스템은 사용자가 기록을 신뢰할 수 있도록 세션 결과를 일관되게 유지할 수 있다.

### 내비게이션 및 앱 구조

- FR28: 사용자는 메인 화면, 주제 관리 화면, 통계 화면 사이를 이동할 수 있다.

- FR30: 사용자는 학습 완료 후 통계 화면으로 이동해 결과를 검토할 수 있다.

### 로컬 데이터 보존

- FR32: 시스템은 세션 기록을 로컬 저장소에 보존할 수 있다.
- FR33: 시스템은 주제 정보와 주간 목표 데이터를 로컬 저장소에 보존할 수 있다.
- FR34: 사용자는 앱을 재실행한 이후에도 기존 기록과 목표 데이터를 확인할 수 있다.
- FR35: 시스템은 네트워크 연결 없이도 핵심 데이터 기반 기능을 사용할 수 있게 할 수 있다.

- FR36: 시스템은 macOS 환경에서 동작하는 데스크톱 앱으로 제공될 수 있다.
- FR37: 시스템은 인터넷 연결 없이도 핵심 학습 흐름을 지원할 수 있다.
- FR38: 시스템은 학습 세션 시작 및 종료와 관련된 기본 알림을 제공할 수 있다.
- FR39: 사용자는 복잡한 계정 생성이나 로그인 없이 앱을 사용할 수 있다.

## Non-Functional Requirements

### Performance

- 메인 화면은 사용자가 앱 실행 직후 오늘 진행 상황과 주간 진척을 빠르게 확인할 수 있을 정도로 지연 없이 표시되어야 한다.
- 주제 선택, 화면 전환, 통계 화면 진입은 사용 흐름을 끊지 않을 정도로 즉각적으로 반응해야 한다.
- 학습 세션 완료 후 기록 저장과 통계 반영은 사용자가 결과를 바로 확인할 수 있도록 즉시 처리되어야 한다.
- 오프라인 상태에서도 핵심 사용자 흐름의 반응 속도는 네트워크 상태에 영향을 받아서는 안 된다.

### Security

- 세션 기록, 주제 정보, 주간 목표 데이터는 사용자의 로컬 기기 내에서 보존되어야 한다.
- 제품은 MVP 범위에서 계정 생성이나 원격 서버 저장을 요구하지 않아야 한다.
- 사용자의 학습 데이터는 의도하지 않은 손실이나 손상 없이 유지되어야 한다.
- 로컬 데이터 처리 방식은 제품 신뢰를 해치지 않도록 단순하고 예측 가능해야 한다.

### Accessibility

- 제품은 기본적인 macOS 접근성 요구를 충족해야 한다.
- 주요 화면과 핵심 흐름은 시각적 정보에만 의존하지 않도록 구성되어야 한다.
- 사용자는 기본적인 시스템 접근성 기능을 활용해 핵심 기능을 사용할 수 있어야 한다.
