## Story

As a 개발자,
I want Tauri 전용 초기화와 분리된 브라우저 검증 경로를 갖추기를,
so that Home과 Session 핵심 흐름을 브라우저에서 빠르게 재현하고 회귀를 확인할 수 있다.

## Acceptance Criteria

1. **Given** 브라우저 런타임에서 앱을 열었을 때 **When** Tauri native API가 존재하지 않으면 **Then** 앱이 bootstrap 단계에서 실패하지 않고 internal QA 검증 경로로 진입한다
2. **And** Tauri runtime에서는 기존 SQLite 초기화, migration, notification 동작이 유지된다
3. **And** 브라우저 검증 경로는 mock/in-memory adapter 또는 동등한 harness를 사용해 Home -> 주제 선택 -> 학습 시작 -> 세션 완료 -> 휴식 시작 흐름을 재현할 수 있다
4. **And** browser 검증 모드는 사용자 제공 플랫폼 범위가 아니라 internal QA 용으로만 노출된다
5. **And** 새로운 runtime seam이 UI에서 Tauri plugin 직접 호출을 늘리지 않는다

### 핵심: 이 스토리는 웹 제품화가 아니라 internal QA runtime 확보다

- PRD는 여전히 macOS 전용 데스크톱 앱을 전제로 한다
- 브라우저 경로는 사용자용 플랫폼이 아니라 QA / 회귀 확인 수단이어야 한다
- 브라우저에서 가능한 것은 Home / Session 핵심 루프 검증이며, 전체 도메인/통계/목표 기능을 모두 웹으로 옮기는 작업이 아니다

### 핵심: 기존 Story 3.4 구현을 다시 만들지 말고 검증 가능하게 연결한다

Story 3.4는 이미 다음을 구현했다.

- study/break phase-aware session lifecycle
- `휴식 시작`, `다음 학습 시작` CTA 분기
- `휴식 진행 중`, `휴식 완료` copy
- break session이 통계에서 제외될 수 있도록 `phaseType` 보존

따라서 이 스토리의 목적은 Story 3.4 기능을 재구현하는 것이 아니라, 브라우저에서도 그 흐름을 확인할 수 있도록 runtime seam을 추가하는 것이다.

### 권장 구현 형태: runtime-aware adapter + 최소 in-memory persistence

권장 방향:

- Tauri path:
  - 기존 `@tauri-apps/plugin-sql` + SQLite + migration 유지
- Browser QA path:
  - Topic / Session에 필요한 최소 in-memory adapter만 제공
  - 동일한 service / store / route를 재사용

권장 이유:

- 현재 repository 계층은 raw SQL을 직접 호출한다
- 브라우저에서 SQL query 문자열을 해석하는 일반화된 mock을 만들면 유지보수 비용이 크다
- Home / Session 검증에는 Topic / Session 도메인 최소 surface만 있으면 충분하다

### Bootstrap 가이드

현재 구조:

- `src/main.tsx` -> `initializeDatabase()` -> `getDb()` -> `Database.load()`

문제:

- 브라우저에는 Tauri native binding이 없어서 bootstrap 단계에서 즉시 실패한다

권장 방향:

```typescript
async function initializeAppRuntime(): Promise<void> {
  if (isTauriRuntime()) {
    await initializeDatabase();
    return;
  }

  await initializeBrowserQaRuntime();
}
```

주의:

- runtime 판별 자체가 브라우저에서 크래시를 유발하지 않도록 safe import / lazy access 전략을 택해야 한다
- 공식 Tauri API의 `core.isTauri` 사용 가능 여부를 우선 검토하되, import 시점 부작용이 없는지 실제 현재 번들에서 확인해야 한다

### Home / Session 연결 가이드

브라우저 QA mode에서 최소한 아래 흐름이 살아 있어야 한다.

1. Home 진입
2. seed topic 표시
3. topic 선택
4. `학습 시작`
5. Session running
6. study 완료
7. `휴식 시작`
8. break running
9. break 완료
10. `다음 학습 시작`

핵심은 `HomeRoute`와 `SessionPage`를 별도 목업 컴포넌트로 복제하지 않는 것이다. 실제 route와 실제 service를 타되, persistence만 runtime seam 뒤에서 바뀌어야 한다.

### Notification / Native 기능 가이드

- `src/platform/tauri/notification-client.ts`는 그대로 유지한다
- 브라우저 QA mode에서는 알림이 핵심 성공 조건이 아니다
- browser path에서 notification을 억지로 지원하려 하지 말고 no-op 또는 graceful skip으로 처리한다

### Testing Guidance

- 새 테스트 프레임워크를 이번 스토리에서 도입하지 않는다
- 기존처럼 `npx tsc --noEmit`, `npm run build`, `node:test` 중심 검증을 우선 사용한다
- 브라우저 flow smoke는 수동 dev-mode 확인 또는 현재 프로젝트 운영 방식에 맞는 경량 스모크 검증으로 제한한다
- Tauri runtime 회귀 확인은 최종적으로 별도 수동 확인이 필요하다

### Anti-Patterns to Avoid

- ❌ 브라우저를 정식 사용자 플랫폼처럼 다루는 구현
- ❌ raw SQL 전부를 브라우저에서 흉내 내는 범용 mock DB 작성
- ❌ UI 컴포넌트에서 `@tauri-apps/plugin-sql` 또는 native API 직접 호출
- ❌ Story 3.4 기능을 별도 browser-only 화면으로 복제
- ❌ Stats / Goals / Records 범위를 이번 스토리에 끌고 들어오기
- ❌ QA mode를 메인 내비게이션이나 제품 기본 경로에 노출하기
- ❌ Tauri bootstrap 경로를 약화시켜 기존 SQLite 동작을 깨뜨리기

### Project Structure Notes

- `src/platform/`은 native/runtime bridge 역할에 맞게 유지한다
- `src/domain/`은 domain model, repository, validation을 유지한다
- `src/features/`는 user-facing route와 service orchestration만 다룬다
- 런타임 선택 로직이 필요해도 `src/app/routes/*`나 컴포넌트에 흩뿌리지 말고 bootstrap 또는 platform 경계에 모은다
