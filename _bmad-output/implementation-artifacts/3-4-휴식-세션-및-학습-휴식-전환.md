# Story 3.4: 휴식 세션 및 학습/휴식 전환

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 학습자,
I want 학습 세션 후 휴식 세션으로 전환할 수 있기를,
so that 학습과 휴식 리듬을 유지할 수 있다.

## Acceptance Criteria

1. **Given** 학습 세션이 완료된 후 **When** 휴식 세션 시작을 선택하면 **Then** 휴식 타이머가 시작된다 (FR5)
2. **And** 시스템이 학습 세션과 휴식 세션을 구분해 기록한다 (FR7)
3. **And** 휴식 세션 완료 시 다음 학습 세션 시작 CTA가 제공된다
4. **And** 휴식 시간은 학습 통계에 포함되지 않는다

### 구현 가정: 기본 휴식 길이는 5분으로 둔다

기획 문서에는 휴식 세션의 정확한 초/분 수치가 명시돼 있지 않다. 다만 PRD가 `기본 포모도로 사이클`을 명시하고 있고, 현재 study 기본값이 25분으로 고정되어 있으므로 이 스토리는 **short break 5분**을 기본값으로 둔다.

- 이 값은 `session-service.ts` 상수 하나로 관리한다
- 장기 휴식(long break), 회차별 다른 휴식 길이, 사용자 설정 UI는 현재 범위 밖이다
- 이후 기획이 바뀌어도 상수 교체만으로 대응 가능하게 구현한다

### 핵심: 휴식 세션은 새 아키텍처가 아니라 기존 세션 모델의 확장이다

현재 코드베이스는 이미 휴식 세션을 받을 준비가 되어 있다.

- `src/domain/sessions/session.ts`의 `SessionPhaseType`은 이미 `'study' | 'break'`를 지원한다
- `src/domain/sessions/session-schema.ts`의 `CreateSessionSchema`도 이미 `phaseType: 'break'`를 허용한다
- `src/db/schema/sessions.sql`과 `src/db/migrations/001_initial_schema.sql`도 `phase_type IN ('study', 'break')`를 사용한다
- `src/domain/sessions/session-mappers.ts`는 DB row의 `phase_type`을 그대로 `Session.phaseType`으로 변환한다

즉, 이 스토리의 핵심은 스키마 추가가 아니라 **UI/서비스 전환 흐름을 정확히 연결하는 것**이다.

### session-service 구현 가이드

현재 `session-service.ts`는 study 전용 함수명으로 작성되어 있지만, 내부 동작은 break에도 그대로 재사용 가능하다. 중복 SQL이나 direct repository 호출을 UI에 퍼뜨리지 말고 서비스 계층에서만 phase 분기를 처리한다.

권장 방향:

```typescript
export const DEFAULT_STUDY_DURATION_SEC = 25 * 60;
export const DEFAULT_BREAK_DURATION_SEC = 5 * 60;

async function beginSessionForPhase(
  topicId: string,
  topicName: string | null,
  phaseType: 'study' | 'break',
  plannedDurationSec: number,
): Promise<Result<Session>> {
  // createSession 재사용
  // 성공 시 selectedTopic 유지 + sessionStore.startSession(result.data)
}

export async function beginBreakSession(...) {
  return beginSessionForPhase(topicId, topicName, 'break', DEFAULT_BREAK_DURATION_SEC);
}
```

완료 처리도 같은 원칙으로 정리한다.

- study 완료 시: 기존 `completeSessionRecord()` + `endSession(result.data)` + optional notification
- break 완료 시: 같은 repository 완료 함수를 쓰되, 학습 완료 알림/카피를 그대로 재사용하지 않는다
- UI 컴포넌트에서 repository를 직접 호출하지 않는다

### SessionPage 전환 가이드

현재 `SessionPage.tsx`는 `sessionPhase === 'completed'`일 때 항상 학습 완료로 간주한다. Story 3.4에서는 completed session의 `phaseType`을 반드시 읽어야 한다.

권장 분기:

```typescript
const completedPhaseType = completedSession?.phaseType ?? activeSession?.phaseType;

if (sessionPhase === 'completed' && completedPhaseType === 'study') {
  // 휴식 시작 CTA
}

if (sessionPhase === 'completed' && completedPhaseType === 'break') {
  // 다음 학습 시작 CTA
}
```

핵심 규칙:

- study 완료 -> primary CTA는 `휴식 시작`
- break 완료 -> primary CTA는 `다음 학습 시작`
- 두 CTA 모두 현재 선택된 topic으로 **직접** 다음 세션을 시작한다
- 직접 시작 실패 시 현재 outcome 화면을 유지하고 에러만 노출한다
- `/session` 라우트를 벗어나지 않으므로 `FocusLayout`과 reduced navigation 패턴은 그대로 유지된다

### SessionOutcomePanel 확장 가이드

`SessionOutcomePanel`은 이미 성공 피드백 UI 뼈대를 갖고 있다. 새로운 phase마다 패널을 복제하지 말고, 아래처럼 copy를 주입받는 재사용 컴포넌트로 일반화한다.

권장 props 확장:

```typescript
interface SessionOutcomePanelProps {
  variant: 'success' | 'recovery';
  topicName: string;
  durationSec: number;
  durationLabel: string; // '학습 완료' | '휴식 완료'
  feedbackMessage: string;
  primaryActionLabel: string;
  onPrimaryAction: () => void;
  onViewStats: () => void;
  onGoHome: () => void;
}
```

이 방식의 장점:

- Story 3.3의 success 화면 자산을 그대로 재사용한다
- Story 3.6의 recovery variant 확장을 막지 않는다
- break 완료에 `학습 완료` 텍스트가 섞이는 회귀를 막는다

### SessionFocusTimer / CharacterStatePanel 가이드

`SessionFocusTimer.tsx`는 현재 상태 문구를 `remainingSec > 0 ? '집중 진행 중' : '세션 완료'`로 고정한다. break session이 들어오면 이 문구가 잘못된다.

따라서:

- `SessionFocusTimer`에 `phaseType` 또는 `statusText` prop을 추가한다
- study running일 때만 `집중 진행 중`
- break running일 때는 `휴식 진행 중`
- `CharacterStatePanel`은 그대로 사용하되 break running에서 `message="휴식 진행 중"`을 넘긴다
- `loading.svg`는 실제 loading indicator가 아니라 session running 상태 전용이다. break에도 session 진행 상태로만 사용한다

### sessionStore 정리 포인트

현재 `sessionStore`는 `startSession()`에서 `completedSession`을 지우지 않는다. Story 3.4에서는 study 완료 -> break 시작 -> break 완료가 같은 화면 흐름 안에서 이어지므로 stale completed data가 남으면 UI 분기가 꼬일 수 있다.

반영 포인트:

- `startSession(session)`에서 `completedSession: null`로 초기화
- `endSession(session)`에서 `completedSession`에 최신 DB row를 저장
- `selectedTopicId` / `selectedTopicName`은 break 전환 동안 유지
- `reset()`은 홈/통계로 나갈 때만 호출

### 데이터/통계 경계 가이드

이 스토리의 AC #4는 현재 없는 통계 화면을 미리 구현하라는 뜻이 아니다. 지금 해야 할 일은 **휴식 세션이 미래 통계 계산에서 배제될 수 있도록 기록을 정확히 남기는 것**이다.

- break session은 반드시 `phaseType: 'break'`로 저장한다
- study progression copy에 break duration을 포함하지 않는다
- Epic 5 Story 5.1이 `phase_type = 'study'`만 집계하도록 설계돼 있으므로, 지금 필요한 것은 phase 구분의 정확한 보존이다
- `features/stats/`, `domain/statistics/`, 대시보드 합계 계산을 이 스토리에서 선행 구현하지 않는다

### Anti-Patterns to Avoid

- ❌ 휴식 세션용 새 DB 테이블/마이그레이션 추가
- ❌ `SessionPage.tsx`나 컴포넌트에서 repository/Tauri plugin 직접 호출
- ❌ `BreakOutcomePanel` 같은 중복 컴포넌트 생성
- ❌ break 완료를 `학습 완료` 문구나 학습 진척으로 표시
- ❌ break 시작 전에 `reset()`을 호출해 topic context를 잃어버리기
- ❌ single-running-session 제약을 우회하는 예외 로직 추가
- ❌ 장기 휴식, 회차 카운터, 사용자 지정 타이머 설정까지 범위 확대
- ❌ 알림 permission 실패를 세션 핵심 흐름 에러로 승격
- ❌ statistics/dashboard 기능을 선행 구현해 Epic 5 범위를 침범

### Previous Story Intelligence

**Story 3.3에서 바로 이어받는 사항:**

- `src/features/session/components/SessionOutcomePanel.tsx`는 현재 success/recovery variant 구조를 갖고 있지만, success copy와 CTA가 학습 완료에 고정되어 있다
- `src/features/session/SessionPage.tsx`의 completed 분기는 `completedSession`을 사용해 실제 duration을 계산한다
- `src/features/session/session-service.ts`는 DB 성공 후에만 store를 갱신하고, study 완료 후 optional notification을 fire-and-forget으로 보낸다
- `src/features/session/state/sessionStore.ts`는 `completedSession`을 이미 갖고 있으므로 새 전환 흐름도 이 저장 방식을 재사용할 수 있다
- Story 3.3의 anti-pattern 중 하나가 `휴식 세션 타이머/전환 — Story 3.4 범위`였으므로, 이제 그 미뤄 둔 부분만 정확히 채우면 된다

**Story 3.2에서 유지해야 할 사항:**

- `useSessionClock.ts`의 timestamp 기반 계산은 phase와 무관하게 그대로 재사용 가능하다
- `SessionFocusTimer.tsx`의 타이머/원형 진행률 UI는 유지하되 문구만 phase-aware로 바꾼다
- `FocusLayout.tsx`를 통한 reduced navigation 패턴은 session flow 전체에서 계속 유지한다
- `CharacterStatePanel.tsx`는 `message` prop을 받을 수 있으므로 break용 새 캐릭터 상태를 만들 필요가 없다

2026-03-19 기준으로 확인한 공식 문서/가이드에서 현재 구현에 직접 영향을 주는 사항:

- Tauri Notification 공식 문서는 여전히 `isPermissionGranted -> requestPermission -> sendNotification` 순서를 권장한다. Story 3.3의 `notification-client.ts` 구현이 이미 이 패턴과 일치하므로 break 관련 알림을 추가하더라도 같은 래퍼를 재사용하면 된다
- Tauri Permission 문서는 플러그인 권한 식별자가 `<name>:default` 패턴을 따른다고 설명한다. 현재 `notification:default` capability 사용은 이 규칙과 일치하므로, Story 3.4에서는 capability surface를 넓히지 않는다
- React Router `useNavigate` 문서는 timed UI / effect 기반 전환에 programmatic navigation을 사용하는 패턴을 허용한다. 세션 종료 후 `/session` 내부 전환과 홈/통계 이동은 현재처럼 `useNavigate()`를 유지하면 된다
- Zustand 가이드는 `set`이 top-level shallow merge를 수행하며 상태는 immutable하게 갱신돼야 한다고 명시한다. 따라서 `sessionStore` 확장도 in-place mutation 없이 top-level field 갱신으로만 처리한다
