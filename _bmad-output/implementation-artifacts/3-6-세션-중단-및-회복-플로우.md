### 핵심: 이 스토리는 기존 interrupted stub을 recovery UX로 교체하는 작업이다

현재 코드베이스에서 세션 중단은 이미 기능적으로 동작한다:

- `session-service.ts`의 `interruptActiveSession()`이 DB에 interrupted 상태를 저장한다
- `sessionStore`의 `interruptCurrentSession()`이 `sessionPhase: 'interrupted'`를 설정한다
- `SessionPage.tsx`의 `sessionPhase === 'interrupted'` 블록이 있지만 **stub 상태**다

**문제점:**
- `interruptCurrentSession()`이 세션 데이터를 저장하지 않으므로 recovery UI에서 중단된 세션 정보를 참조할 수 없다
- interrupted 블록이 `CharacterStatePanel(state="default")` + 단일 "홈으로 돌아가기" 버튼만 렌더링한다
- recovery variant가 UX-DR5/UX-DR9의 요구사항을 충족하지 않는다

**이 스토리의 핵심은:** stub을 완전한 recovery variant `SessionOutcomePanel`로 교체하고, 회복 가능성 중심의 3-CTA 패턴을 구현하는 것이다.

### sessionStore 수정 가이드

현재 `interruptCurrentSession`은 파라미터 없이 `sessionPhase`만 변경한다:

```typescript
// 현재 (문제)
interruptCurrentSession: () =>
  set({ sessionPhase: 'interrupted' }),
```

수정 방향:

```typescript
// 수정 후
interruptCurrentSession: (session: Session) =>
  set({ sessionPhase: 'interrupted', completedSession: session }),
```

주의: `activeSession`은 그대로 유지한다. `reset()`을 호출할 때까지 topic context, session data가 모두 살아 있어야 recovery UI에서 "바로 재시작"이 가능하다.

### session-service.ts 수정 가이드

현재 `interruptActiveSession`은 `interruptCurrentSession()`에 데이터를 전달하지 않는다:

```typescript
// 현재 (문제)
if (result.ok) {
  useSessionStore.getState().interruptCurrentSession();
}
```

수정 방향:

```typescript
// 수정 후
if (result.ok) {
  useSessionStore.getState().interruptCurrentSession(result.data);
}
```

### session-flow.ts 확장 가이드

recovery 전용 콘텐츠를 추가한다. 기존 `getOutcomeContent()`는 success(완료) 전용이므로 별도 함수를 만든다:

```typescript
interface InterruptedOutcomeContent {
  feedbackMessage: string;
  recoveryHint: string;
  primaryActionLabel: string;
  secondaryActionLabel: string;
  tertiaryActionLabel: string;
}

export function getInterruptedOutcomeContent(): InterruptedOutcomeContent {
  return {
    feedbackMessage: '괜찮아요, 이번 주 안에 다시 이어갈 수 있어요.',
    recoveryHint: '중단해도 괜찮아요. 언제든 다시 시작할 수 있습니다.',
    primaryActionLabel: '바로 재시작',
    secondaryActionLabel: '다른 주제 선택',
    tertiaryActionLabel: '오늘은 종료',
  };
}
```

### SessionOutcomePanel recovery variant 가이드

현재 `variant: 'success' | 'recovery'` prop이 이미 정의되어 있지만 `_variant`로 무시되고 있다. recovery variant에서 달라지는 점:

1. **캐릭터 상태**: `speak.svg` 유지 (동일) — 회복 메시지 전달
2. **요약 영역**: `durationLabel`이 "진행 중단"으로 변경됨
3. **힌트 텍스트**: 추가 요소 — "이번 주 기준 남은 목표" 안내 문구
4. **CTA 구성**: 3개 — "바로 재시작" (primary), "다른 주제 선택" (secondary), "오늘은 종료" (text)
5. **"통계 보기" CTA**: recovery에서는 미표시

권장 방향: `variant`에 따라 CTA 영역을 조건부 렌더링한다.

```tsx
{variant === 'success' ? (
  <>
    <Button variant="primary" ...>{primaryActionLabel}</Button>
    <Button variant="secondary" onClick={onViewStats} ...>통계 보기</Button>
    <Button variant="text" onClick={onGoHome} ...>홈으로</Button>
  </>
) : (
  <>
    <Button variant="primary" ...>{primaryActionLabel}</Button>
    <Button variant="secondary" onClick={onSelectOtherTopic} ...>다른 주제 선택</Button>
    <Button variant="text" onClick={onGoHome} ...>오늘은 종료</Button>
  </>
)}
```

### SessionPage interrupted 블록 교체 가이드

현재 stub (SessionPage.tsx 171-187행):

```tsx
if (sessionPhase === 'interrupted') {
  return (
    <div className={styles.page}>
      <div className={styles.endState}>
        <CharacterStatePanel state="default" message="세션이 중단되었습니다" />
        <p className={styles.endMessage}>세션이 중단되었습니다</p>
        <button className={styles.homeButton} onClick={handleGoHome} type="button">
          홈으로 돌아가기
        </button>
      </div>
    </div>
  );
}
```

교체 방향:

```tsx
if (sessionPhase === 'interrupted') {
  const interruptedContent = getInterruptedOutcomeContent();
  const interruptedDurationSec = (() => {
    const session = completedSession ?? activeSession;
    if (session?.endedAtMs && session.startedAtMs) {
      return Math.floor((session.endedAtMs - session.startedAtMs) / 1000);
    }
    return activeSession?.plannedDurationSec ?? 0;
  })();

  return (
    <div className={styles.page}>
      <SessionOutcomePanel
        variant="recovery"
        topicName={selectedTopicName ?? ''}
        durationSec={interruptedDurationSec}
        durationLabel="진행 중단"
        feedbackMessage={interruptedContent.feedbackMessage}
        primaryActionLabel={interruptedContent.primaryActionLabel}
        onPrimaryAction={handleStartNextStudy}
        onViewStats={handleViewStats}
        onGoHome={handleGoHome}
        isBusy={isSaving}
        recoveryHint={interruptedContent.recoveryHint}
        onSelectOtherTopic={handleGoHome}
      />
      {actionError && (
        <p className={styles.actionError} role="alert">
          {actionError}
        </p>
      )}
    </div>
  );
}
```

### AC #4 "이번 주 기준 남은 목표" — 현재 범위 제한

Epic 4 (주간 목표 설정)와 Epic 5 (통계)는 아직 구현되지 않았다. 따라서 이번 스토리에서 "이번 주 기준 남은 목표"를 실제 데이터로 표시할 수 없다.

현재 범위의 구현:
- 정적 힌트 문구로 대체: "중단해도 괜찮아요. 언제든 다시 시작할 수 있습니다."
- Epic 4/5 구현 후에 실데이터 연동 가능하도록 `recoveryHint` prop으로 분리한다
- `features/stats/`, `domain/statistics/`, `domain/goals/` 모듈에 의존하는 코드를 이 스토리에서 작성하지 않는다

### Anti-Patterns to Avoid

- ❌ stats/goals 모듈에 의존하는 recovery 데이터 조회 (Epic 4/5 범위)
- ❌ `SessionOutcomePanel` 대신 별도 `RecoveryPanel` 컴포넌트 생성
- ❌ `interruptCurrentSession()`에서 `activeSession`을 null로 세팅해 topic context를 잃어버리기
- ❌ interrupted 상태에서 "통계 보기"를 primary CTA로 두기
- ❌ recovery 메시지에 실패감을 주는 표현 사용 ("실패", "미달성", "부족" 등)
- ❌ UI 컴포넌트에서 repository/Tauri plugin 직접 호출
- ❌ `handleGoHome` 대신 직접 `navigate()`를 호출해 store reset을 빠뜨리기
- ❌ break 중단 시 다른 recovery 화면을 만들기 (study/break 중단 모두 같은 recovery variant 사용)

### Previous Story Intelligence

**Story 3.4에서 이어받는 핵심:**

- `session-flow.ts`의 `getOutcomeContent()`는 study/break 완료(success) 전용이다. recovery용은 별도 함수로 추가한다
- `SessionOutcomePanel.tsx`의 `variant` prop이 이미 `'success' | 'recovery'`를 허용하지만, `_variant`로 무시되고 있다
- SessionPage의 interrupted 블록 주석에 이미 "Story 3.5에서 recovery variant로 교체 예정"이라고 명시되어 있다 (실제로는 3.6에서 처리)
- `sessionStore`의 `startSession()`은 `completedSession: null`로 초기화하므로, "바로 재시작" 후에 이전 interrupted 데이터가 자연스럽게 정리된다

**Story 3.5에서 이어받는 핵심:**

- Browser QA mode는 `interruptActiveSession()`을 포함하여 in-memory adapter에서 완전 동작한다
- 브라우저에서 study running → 중단 → interrupted 상태 전이가 작동한다
- 따라서 이 스토리의 recovery UI도 브라우저 QA mode에서 수동 검증할 수 있다

**현재 코드에서 확인된 잠재 회귀 포인트:**

- `interruptCurrentSession()` 시그니처 변경은 기존 `session-service.ts` 호출부만 수정하면 된다 (유일한 호출처)
- `SessionOutcomePanel`에 optional prop을 추가하므로 기존 success variant 호출은 영향받지 않는다
- 기존 `session-store.test.mjs`에서 `interruptCurrentSession()` 호출 시 인자를 추가해야 할 수 있다
