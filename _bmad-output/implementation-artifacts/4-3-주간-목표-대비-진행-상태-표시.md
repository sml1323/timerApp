## Story

As a 학습자,
I want 주제별 주간 목표 대비 진행 상태를 확인할 수 있기를,
so that 목표를 달성하지 못한 날에도 주간 기준으로 남은 목표를 파악하고 이어갈 수 있다.

2. **And** 목표 미달 시에도 "이번 주 남은 목표" 형태로 표시된다 (FR17)
3. **And** 회복 중심 문구가 사용된다 (UX-DR9)
4. **And** 진행률은 `sessions` 테이블에서 파생 계산된다

### 아키텍처 패턴 — 반드시 따를 것

#### 1. 세션 통계 쿼리 (`session-statistics.ts`)

이 파일은 Architecture 문서 `src/domain/sessions/session-statistics.ts` (line 622)에 명시되어 있으나 아직 생성되지 않았다. **집계 쿼리를 직접 SQL로 구현하여 성능을 확보한다** (Architecture: "무거운 집계는 SQL 쿼리에서 처리").

```typescript
// src/domain/sessions/session-statistics.ts
import { select } from '../../platform/tauri/sql-client';
import { ok, err, type Result } from '../../shared/lib/result';
import { ERROR_CODES } from '../../shared/lib/errors';

/** 주간 주제별 학습 누적 분 */
export interface WeeklyStudyByTopic {
  topicId: string;
  totalMinutes: number;
}

/**
 * 특정 주 범위의 completed study 세션을 주제별로 그룹화하여 총 학습 분을 반환한다.
 * - phase_type = 'study' (휴식 제외)
 * - status = 'completed' (interrupted/running 제외)
 * - started_at_ms 범위: [weekStartAtMs, weekStartAtMs + 7일)
 */
export async function getWeeklyStudyMinutesByTopic(
  weekStartAtMs: number
): Promise<Result<Map<string, number>>> {
  try {
    const weekEndAtMs = weekStartAtMs + 7 * 24 * 60 * 60 * 1000;
    const rows = await select<{ topic_id: string; total_seconds: number }>(
      `SELECT topic_id, SUM(planned_duration_sec) AS total_seconds
       FROM sessions
       WHERE phase_type = 'study'
         AND status = 'completed'
         AND started_at_ms >= $1
         AND started_at_ms < $2
       GROUP BY topic_id`,
      [weekStartAtMs, weekEndAtMs],
    );

    const map = new Map<string, number>();
    for (const row of rows) {
      map.set(row.topic_id, Math.round(row.total_seconds / 60));
    }
    return ok(map);
  } catch (error) {
    return err(
      ERROR_CODES.PERSISTENCE_ERROR,
      `주간 학습 통계 조회 중 오류: ${error instanceof Error ? error.message : String(error)}`,
    );
  }
}
```

⚠️ **중요 설계 결정:**
- `planned_duration_sec`(계획된 시간)를 사용 — 실제 경과 시간(`endedAtMs - startedAtMs`)이 아님. 포모도로 완료 = 계획 시간 달성이므로 `planned_duration_sec`가 더 정확함
- `status = 'completed'`만 포함 — 중단된 세션은 목표 진행에 미포함
- SQL 집계로 성능 확보 — 전체 세션을 TS로 가져와 클라이언트에서 집계하지 않음

#### 2. In-Memory Adapter 확장 패턴

`in-memory-session-adapter.ts`의 기존 `store: Session[]`을 활용하여 동일 로직 구현:

```typescript
// in-memory-session-adapter.ts에 추가
export async function getWeeklyStudyMinutesByTopic(
  weekStartAtMs: number
): Promise<Result<Map<string, number>>> {
  const weekEndAtMs = weekStartAtMs + 7 * 24 * 60 * 60 * 1000;
  const map = new Map<string, number>();

  for (const s of store) {
    if (
      s.phaseType === 'study' &&
      s.status === 'completed' &&
      s.startedAtMs >= weekStartAtMs &&
      s.startedAtMs < weekEndAtMs
    ) {
      const mins = Math.round(s.plannedDurationSec / 60);
      map.set(s.topicId, (map.get(s.topicId) ?? 0) + mins);
    }
  }
  return ok(map);
}
```

### Anti-Patterns to Avoid

- ❌ `domain/statistics/` 에 파일 생성 — Epic 5 범위. 이 스토리의 세션 집계는 `domain/sessions/session-statistics.ts`에 둠
- ❌ 통계 결과를 별도 테이블/캐시에 저장 — sessions가 단일 사실원천, 매번 파생 계산
- ❌ 전체 세션을 TS로 가져와 클라이언트에서 집계 — SQL에서 GROUP BY 처리
- ❌ `interrupted` 세션을 진행률에 포함 — `completed` 학습 세션만
- ❌ 휴식(`break`) 세션을 진행률에 포함 — `study` 세션만
- ❌ 목표 미달 시 "실패", "미달성", 경고 빨강 사용 — 회복 중심 문구만 (UX-DR9)
- ❌ `GoalProgressInline`에서 repository 직접 호출 — service → adapter → repository 경로만
- ❌ `useGoalProgress` 안에서 직접 SQL 호출 — hook → service → adapter 경로
- ❌ Zustand store에 진행률 캐시 — 로컬 상태로 충분
- ❌ `TopicQuickSelectPanel` 컴포넌트 안에서 progress 계산 — 훅에서 계산된 데이터만 prop으로 받음
- ❌ `endedAtMs - startedAtMs`로 실 학습 시간 계산 — `plannedDurationSec` 사용 (포모도로 완료 = 계획 시간 달성)
- ❌ `tokens.css` 변수 대신 하드코딩 색상/간격 사용
- ❌ 프로그레스 바에 색상만으로 상태 전달 — 수치/텍스트 반드시 동반 (UX-DR11)
