## Story

As a 학습자,
I want 학습 세션 데이터가 안정적으로 저장되기를,
so that 세션 기록이 통계의 신뢰 가능한 단일 사실원천이 된다.

## Acceptance Criteria

1. **Given** `sessions` 테이블이 DB에 존재할 때 **When** 세션 도메인 모듈을 구현하면 **Then** `session.ts`에 Session 타입이 정의된다 (id, topicId, phaseType, status, startedAtMs, plannedDurationSec, endedAtMs)
2. **And** `session-schema.ts`에 Zod 검증 스키마가 정의된다
3. **And** `session-repository.ts`에 create, complete, interrupt, findByDateRange 함수가 구현된다
4. **And** `session-mappers.ts`에 DB ↔ TS 매핑이 구현된다
5. **And** `session-transitions.ts`에 상태 전이 규칙이 정의된다 (planned → running → completed/interrupted)
6. **And** 타이머는 timestamp 기반(`startedAtMs`, `plannedDurationSec`, `endedAtMs`)으로 모델링된다
7. **And** 모든 함수가 discriminated result 객체를 반환한다

### ⚠️ 핵심: sessions 테이블 스키마 (이미 존재)

```sql
CREATE TABLE IF NOT EXISTS sessions (
    id TEXT PRIMARY KEY NOT NULL,
    topic_id TEXT NOT NULL,
    phase_type TEXT NOT NULL CHECK (phase_type IN ('study', 'break')),
    status TEXT NOT NULL CHECK (status IN ('planned', 'running', 'completed', 'interrupted')),
    started_at_ms INTEGER NOT NULL,
    planned_duration_sec INTEGER NOT NULL,
    ended_at_ms INTEGER,
    created_at_ms INTEGER NOT NULL,
    updated_at_ms INTEGER NOT NULL,
    FOREIGN KEY (topic_id) REFERENCES topics(id) ON DELETE RESTRICT
);
```

### ⚠️ 기존 topic-repository.ts 패턴을 100% 따를 것

`topic-repository.ts`에서 확립된 패턴을 세션 repository에 동일하게 적용한다:

1. **import 구조**: `select`, `execute` from sql-client → `ok`, `err`, `Result` from result.ts → `ERROR_CODES` from errors.ts
2. **Zod 검증**: `Schema.safeParse(input)` → 실패 시 `err(ERROR_CODES.VALIDATION_ERROR, message)` 반환
3. **ID 생성**: `crypto.randomUUID()`
4. **타임스탬프**: `Date.now()`
5. **INSERT 후 조회**: INSERT → `SELECT * WHERE id = $1` → `toSession(row)` 반환
6. **UPDATE 후 조회**: UPDATE → `SELECT * WHERE id = $1` → `toSession(row)` 반환
7. **에러 래핑**: 모든 함수 최외곽 `try/catch` → `err(ERROR_CODES.PERSISTENCE_ERROR, message)`
8. **파라미터 바인딩**: `$1`, `$2`, ... 형식 (Tauri SQL 플러그인 규칙)

```typescript
// topic-repository.ts에서 입증된 패턴 예시:
export async function createTopic(input: CreateTopicInput): Promise<Result<Topic>> {
  try {
    const parsed = CreateTopicSchema.safeParse(input);
    if (!parsed.success) {
      return err(ERROR_CODES.VALIDATION_ERROR, parsed.error.issues[0]?.message ?? '...');
    }
    const id = crypto.randomUUID();
    const now = Date.now();
    await execute('INSERT INTO topics ...', [id, name, ...]);
    const rows = await select<TopicRow>('SELECT * FROM topics WHERE id = $1', [id]);
    return ok(toTopic(rows[0]));
  } catch (error) {
    return err(ERROR_CODES.PERSISTENCE_ERROR, `... ${error instanceof Error ? error.message : String(error)}`);
  }
}
```

### Session Mappers 가이드

```typescript
// src/domain/sessions/session-mappers.ts
import type { Session } from './session';

export interface SessionRow {
  id: string;
  topic_id: string;
  phase_type: string;        // 'study' | 'break'
  status: string;            // 'planned' | 'running' | 'completed' | 'interrupted'
  started_at_ms: number;
  planned_duration_sec: number;
  ended_at_ms: number | null;
  created_at_ms: number;
  updated_at_ms: number;
}

export function toSession(row: SessionRow): Session {
  return {
    id: row.id,
    topicId: row.topic_id,
    phaseType: row.phase_type as Session['phaseType'],
    status: row.status as Session['status'],
    startedAtMs: row.started_at_ms,
    plannedDurationSec: row.planned_duration_sec,
    endedAtMs: row.ended_at_ms,
    createdAtMs: row.created_at_ms,
    updatedAtMs: row.updated_at_ms,
  };
}
```

> **주의**: `ended_at_ms`는 nullable이다. DB에서 NULL로 반환될 수 있으며 `running`/`planned` 상태에서는 null이 정상이다.

### Session Transitions 가이드

```typescript
// src/domain/sessions/session-transitions.ts
import type { SessionStatus } from './session';
import { ok, err, type Result } from '../../shared/lib/result';
import { ERROR_CODES } from '../../shared/lib/errors';

const VALID_TRANSITIONS: Record<SessionStatus, SessionStatus[]> = {
  planned: ['running'],
  running: ['completed', 'interrupted'],
  completed: [],
  interrupted: [],
};

export function canTransition(from: SessionStatus, to: SessionStatus): boolean {
  return VALID_TRANSITIONS[from]?.includes(to) ?? false;
}

export function validateTransition(from: SessionStatus, to: SessionStatus): Result<void> {
  if (canTransition(from, to)) {
    return ok(undefined);
  }
  return err(
    ERROR_CODES.SESSION_STATE_CONFLICT,
    `세션 상태를 '${from}'에서 '${to}'로 변경할 수 없습니다`,
  );
}
```

### createSession 구현 핵심 로직

```
1. Zod 검증 (CreateSessionSchema)
2. topicId 존재 확인: SELECT id FROM topics WHERE id = $1
   → 없으면 err(NOT_FOUND, '주제를 찾을 수 없습니다')
3. 현재 running 세션 존재 확인: SELECT id FROM sessions WHERE status = 'running' LIMIT 1
   → 있으면 err(SESSION_STATE_CONFLICT, '이미 진행 중인 세션이 있습니다')
4. INSERT (status='running', startedAtMs=Date.now())
   → 세션 생성 시 바로 running 상태로 시작 (planned 상태 건너뜀 — MVP 단순화)
5. SELECT * WHERE id = $1 → toSession(row) 반환
```

> **설계 결정**: MVP에서는 `planned` 상태 없이 세션 생성 즉시 `running` 상태로 시작한다. DB 스키마에 `planned`가 정의되어 있으나, 현재 흐름은 주제 선택 → 시작 CTA → 세션 생성(=running)이므로 `planned` 단계가 불필요하다. `session-transitions.ts`에는 `planned → running` 전이를 정의해두어 향후 확장에 대비한다.

### completeSession / interruptSession 구현 핵심 로직

```
1. Zod 검증 (sessionId)
2. findSessionById로 현재 세션 조회
   → 없으면 err(NOT_FOUND, '세션을 찾을 수 없습니다')
3. validateTransition(session.status, 'completed' 또는 'interrupted')
   → 실패 시 SESSION_STATE_CONFLICT 에러 반환
4. UPDATE sessions SET status = $1, ended_at_ms = $2, updated_at_ms = $3 WHERE id = $4
5. SELECT * WHERE id = $1 → toSession(row) 반환
```

### findSessionsByDateRange 구현 가이드

```sql
SELECT * FROM sessions
WHERE started_at_ms >= $1 AND started_at_ms < $2
ORDER BY started_at_ms DESC
```

- 파라미터: `startMs` (포함), `endMs` (미포함)
- `status` 필터는 선택적 파라미터로 두되, 기본 전달 없으면 모든 상태 반환
- 향후 통계 계산에서 `completed` + `phase_type = 'study'`만 필터링하는 것은 통계 도메인의 책임

### Anti-Patterns to Avoid

- ❌ 새 마이그레이션 파일 생성 — sessions 테이블은 이미 존재
- ❌ `sql-client.ts` 수정 — 기존 `select`, `execute` 그대로 사용
- ❌ React 컴포넌트/훅 생성 — 이 스토리는 도메인 계층만 (UI는 Story 3.2에서)
- ❌ Zustand store 생성 — 이 스토리는 repository 계층만 (store는 Story 3.2에서)
- ❌ `features/session/` 파일 수정 — 이 스토리 범위 밖
- ❌ `topic-repository.ts` 직접 import — topics 존재 확인은 SQL 직접 조회 (`SELECT id FROM topics WHERE id = $1`)
- ❌ `session-service.ts` 생성 — service 계층은 Story 3.2에서 생성
- ❌ incrementing counter 방식 타이머 — timestamp 기반만 사용
- ❌ `result.ts`, `errors.ts` 수정 — 기존 타입과 코드 그대로 재사용
- ❌ Tailwind, styled-components, inline styles
- ❌ `loading.svg`를 로딩 상태로 사용 — `loading.svg`는 세션 진행 중 UI 전용 (Story 3.2)
- ❌ 통계 계산 로직 — 통계는 Epic 5 범위
- ❌ `ended_at_ms`를 0이나 -1 대신 null로 처리하지 않는 것 — 반드시 nullable (DB 컬럼이 `INTEGER` nullable)
