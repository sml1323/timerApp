Status: done

## Story

As a 학습자,
I want 주간 목표 데이터가 안정적으로 저장되기를,
so that 앱 재실행 후에도 목표가 보존된다.

## Acceptance Criteria

1. **Given** `weekly_goals` 테이블이 DB에 존재할 때 **When** 목표 도메인 모듈을 구현하면 **Then** `weekly-goal.ts`에 WeeklyGoal 타입이 정의된다 (id, topicId, weekStartAtMs, targetMinutes)
2. **And** `weekly-goal-schema.ts`에 Zod 검증 스키마가 정의된다 (양수 목표 분, 유효한 주 시작일)
3. **And** `weekly-goal-repository.ts`에 create, update, findByTopicAndWeek 함수가 구현된다
4. **And** `weekly-goal-mappers.ts`에 DB ↔ TS 매핑이 구현된다
5. **And** 모든 함수가 discriminated result 객체를 반환한다

### DB 스키마 — 이미 존재함

`weekly_goals` 테이블은 `001_initial_schema.sql`에서 이미 생성되어 있다:

```sql
CREATE TABLE IF NOT EXISTS weekly_goals (
    id TEXT PRIMARY KEY NOT NULL,
    topic_id TEXT NOT NULL,
    week_start_at_ms INTEGER NOT NULL,
    target_minutes INTEGER NOT NULL,
    created_at_ms INTEGER NOT NULL,
    updated_at_ms INTEGER NOT NULL,
    FOREIGN KEY (topic_id) REFERENCES topics(id) ON DELETE RESTRICT
);
```

복합 인덱스도 이미 존재한다:
```sql
CREATE INDEX IF NOT EXISTS idx_weekly_goals_topic_id_week_start_at_ms
    ON weekly_goals(topic_id, week_start_at_ms);
```

**⚠️ 새 마이그레이션 파일을 생성하지 말 것! DB 스키마와 인덱스는 모두 준비되어 있다.**

### TypeScript 타입 설계 가이드

```typescript
// weekly-goal.ts
export interface WeeklyGoal {
  id: string;
  topicId: string;
  weekStartAtMs: number;   // 해당 주 월요일 00:00 epoch ms
  targetMinutes: number;   // 양수 정수
  createdAtMs: number;
  updatedAtMs: number;
}

export interface CreateWeeklyGoalInput {
  topicId: string;
  weekStartAtMs: number;
  targetMinutes: number;
}

export interface UpdateWeeklyGoalInput {
  targetMinutes: number;
}
```

### Mapper 설계 가이드

```typescript
// weekly-goal-mappers.ts — topic-mappers.ts 패턴 그대로 따름
export interface WeeklyGoalRow {
  id: string;
  topic_id: string;
  week_start_at_ms: number;
  target_minutes: number;
  created_at_ms: number;
  updated_at_ms: number;
}

export function toWeeklyGoal(row: WeeklyGoalRow): WeeklyGoal {
  return {
    id: row.id,
    topicId: row.topic_id,
    weekStartAtMs: row.week_start_at_ms,
    targetMinutes: row.target_minutes,
    createdAtMs: row.created_at_ms,
    updatedAtMs: row.updated_at_ms,
  };
}
```

### Zod 스키마 설계 가이드

```typescript
// weekly-goal-schema.ts
import { z } from 'zod';

export const CreateWeeklyGoalSchema = z.object({
  topicId: z.string().min(1, '주제 ID가 필요합니다'),
  weekStartAtMs: z.number().int().positive('유효한 주 시작 시각이 필요합니다'),
  targetMinutes: z.number().int().min(1, '목표 시간은 1분 이상이어야 합니다'),
});

export const UpdateWeeklyGoalSchema = z.object({
  targetMinutes: z.number().int().min(1, '목표 시간은 1분 이상이어야 합니다'),
});
```

### Repository 설계 가이드

`topic-repository.ts` 패턴을 정확히 따른다:

1. **import 구조**: `select`/`execute` from `../../platform/tauri/sql-client`, `ok`/`err` from `../../shared/lib/result`, `ERROR_CODES` from `../../shared/lib/errors`
2. **Zod 검증 우선**: `safeParse` 후 실패 시 `VALIDATION_ERROR` 반환
3. **중복 방지**: `createWeeklyGoal`에서 동일 `topic_id + week_start_at_ms` 조합 존재 확인
4. **ID 생성**: `crypto.randomUUID()`
5. **타임스탬프**: `Date.now()`로 `created_at_ms`, `updated_at_ms` 설정
6. **INSERT 후 재조회**: INSERT → SELECT 로 생성된 row 반환
7. **에러 래핑**: 모든 함수 try/catch, `PERSISTENCE_ERROR`로 감싸기
8. **SQL 파라미터**: `$1`, `$2`, ... 플레이스홀더 사용

`findByTopicAndWeek` 특이사항:
- 결과가 없으면 에러가 아니라 `ok(null)` 반환 — topic의 `findTopicById`와 달리, 목표가 없는 상태는 정상이다
- Story 4.2에서 "이미 설정된 목표가 있으면 현재 값이 표시된다"를 구현할 때 이 함수를 사용한다

`findAllByWeek` 추가 함수:
- 특정 주의 모든 topic 목표를 한 번에 조회한다
- Story 4.3에서 홈/대시보드에 진행률을 표시할 때 필요하다
- `ok([])` 반환 시 빈 배열 — 에러가 아님

### 주간 시작 시각(weekStartAtMs) 결정 규칙

- `weekStartAtMs`는 해당 주 월요일 00:00:00 UTC의 epoch milliseconds
- 이 스토리에서는 `weekStartAtMs` 값을 **호출자가 계산해서 전달**한다 — repository는 받은 값을 저장만 한다
- `weekStartAtMs` 계산 유틸리티는 `src/shared/lib/dates.ts`에 추가할 수 있지만, 이 스토리의 필수 범위가 아니다
- Story 4.2에서 UI가 현재 주의 `weekStartAtMs`를 계산해 전달한다

### Anti-Patterns to Avoid

- ❌ 새 마이그레이션 파일 생성 (테이블/인덱스 이미 존재)
- ❌ `findByTopicAndWeek`에서 결과 없을 때 `NOT_FOUND` 에러 반환 — `ok(null)` 이 올바른 패턴
- ❌ repository 내에서 `weekStartAtMs` 계산 (호출자 책임)
- ❌ UI 컴포넌트나 feature 서비스를 이 스토리에서 구현
- ❌ `camelCase` DB 컬럼명 사용 (반드시 `snake_case`)
- ❌ `snake_case` TypeScript 프로퍼티 사용 (반드시 `camelCase`)
- ❌ mapper 외부에서 DB row를 직접 TS 객체로 다루기
- ❌ `ok` / `err` 대신 raw throw 사용
- ❌ Zustand store에 weekly goal 데이터를 캐시하기 (Story 4.2 범위, 이 스토리에서는 불필요)
- ❌ `domain/goals/` 대신 다른 위치에 파일 생성
