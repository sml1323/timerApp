## Story

1. **Given** 세션이 완료되어 기록이 저장된 후 **When** 홈 화면(`/`)으로 이동하면 **Then** Study Status Summary Card의 오늘 누적, 주간 진척이 즉시 갱신되어 표시된다 (FR23)

3. **And** 통계 재계산은 persisted records(SQLite/in-memory)에서 수행된다
4. **And** 학습 시작 전 주제 관리 화면 진입이 가능하다 (FR29)
5. **And** 학습 완료 후 통계 화면으로 직접 이동해 결과를 검토할 수 있다 (FR30)

## Dev Notes

### 핵심 문제 분석

| 항목 | 상세 |
|------|------|
| **문제** | `useDashboardData`와 `useStatistics` 훅은 `useEffect` + `useCallback(fn, [])` 패턴으로 **마운트 시 1회만** 데이터를 가져옴. React Router가 라우트 전환 시 컴포넌트를 다시 마운트하면 `useEffect`가 다시 실행되지만, **세션 완료와 네비게이션 사이의 타이밍에 따라 stale 데이터가 표시될 수 있음** |
| **근본 원인** | 데이터 훅에 "세션이 방금 완료되었으니 데이터를 다시 가져와야 한다"는 신호가 없음 |
| **해법** | `sessionStore`에 `lastCompletedAtMs` 타임스탬프를 추가하고, 데이터 훅들이 이 값의 변경을 감지하면 `fetchData()`를 재호출하도록 함 |
| **대안 고려** | Event bus, React context 최신 값 전파 등도 가능하지만, 아키텍처 원칙상 Zustand store를 통한 간단한 상태 공유가 가장 적합 |

### 아키텍처 패턴 — 반드시 따를 것

#### 1. sessionStore 변경 (최소 침습)

```typescript
// src/features/session/state/sessionStore.ts — 변경 부분만
interface SessionState {
  // ... 기존 필드 유지
  lastCompletedAtMs: number | null;  // ← 추가
  // ... 기존 액션 유지
}

const initialState = {
  // ... 기존 필드 유지
  lastCompletedAtMs: null,           // ← 추가
};

// endSession 수정
endSession: (session: Session) =>
  set({
    activeSession: session,
    sessionPhase: 'completed',
    completedSession: session,
    lastCompletedAtMs: Date.now(),    // ← 추가
  }),

// interruptCurrentSession 수정
interruptCurrentSession: (session: Session) =>
  set({
    sessionPhase: 'interrupted',
    completedSession: session,
    lastCompletedAtMs: Date.now(),    // ← 추가
  }),

// reset은 lastCompletedAtMs를 유지해야 함 — 중요!
// SessionPage에서 handleGoHome/handleViewStats가 reset() → navigate() 순서로 호출
// reset() 시 lastCompletedAtMs를 null로 만들면 네비게이션 목적지의 훅이 갱신 신호를 놓침
reset: () => set({
  ...initialState,
  lastCompletedAtMs: useSessionStore.getState().lastCompletedAtMs,
}),
```

⚠️ **`reset()`에서 `lastCompletedAtMs`를 유지하는 이유:**
- `SessionPage.handleGoHome()`: `reset()` → `navigate('/')` 순서
- `SessionPage.handleViewStats()`: `reset()` → `navigate('/stats')` 순서
- `reset()`이 `lastCompletedAtMs`를 `null`로 만들면, 목적지 컴포넌트의 `useEffect`가 `null → null` 변경을 감지하지 못해 refetch하지 않음
- `lastCompletedAtMs`는 다음 세션 시작 시 `startSession()`에서 자연스럽게 이전 값으로 남아있어도 문제없음 (훅은 값 변경만 감지)

#### 2. useDashboardData 훅 수정

```typescript
// src/features/dashboard/hooks/useDashboardData.ts
import { useState, useEffect, useCallback } from 'react';
import { loadDashboardData, type DashboardData } from '../dashboard-service';
import { useSessionStore } from '../../session/state/sessionStore';

export function useDashboardData() {
  const lastCompletedAtMs = useSessionStore((state) => state.lastCompletedAtMs);
  const [data, setData] = useState<DashboardData | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  const fetchData = useCallback(async () => {
    setIsLoading(true);
    setError(null);
    const result = await loadDashboardData();
    if (result.ok) {
      setData(result.data);
    } else {
      setError(result.message);
    }
    setIsLoading(false);
  }, []);

  useEffect(() => {
    fetchData();
  }, [fetchData, lastCompletedAtMs]); // ← lastCompletedAtMs 의존성 추가

  return { data, isLoading, error, refetch: fetchData };
}
```

### Anti-Patterns to Avoid

- ❌ Event bus나 커스텀 이벤트 시스템 도입 — Zustand의 구독 기반이 아키텍처에 더 적합
- ❌ `completeActiveSession()` 내부에서 통계 데이터를 미리 fetch — UI 계층의 관심사를 service에 넣지 말 것
- ❌ `HomeRoute`나 `StatsRoute`에서 직접 sessionStore를 구독하고 수동으로 refetch 호출 — 훅 내부에서 처리해야 컴포넌트가 단순하게 유지됨
- ❌ `useEffect` 의존 배열에 `fetchData`와 `lastCompletedAtMs` 외 다른 값 추가 — 불필요한 refetch 방지
- ❌ `lastCompletedAtMs`를 `Date.now()` 대신 `boolean` 플래그로 구현 — 같은 세션 완료 시 여러 번 toggle 필요하면 boolean은 `true→true` 변경을 감지하지 못함
- ❌ `reset()`에서 `lastCompletedAtMs`를 `null`로 초기화 — 네비게이션 목적지 훅이 갱신 신호를 놓침
- ❌ 새로운 Zustand store 생성 — `sessionStore`에 캡슐화가 적절함
- ❌ `useStatistics`와 `useDashboardData`를 하나로 합침 — 기존 feature 분리 원칙 유지
- ❌ 통계 데이터를 Zustand store에 캐싱 — 아키텍처 원칙: persisted data는 SQLite에서 re-read, Zustand는 active UI state만
